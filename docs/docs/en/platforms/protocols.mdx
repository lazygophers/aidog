# Supported Platforms & Interfaces

AiDog supports a wide range of AI platforms and interface types, automatically handling protocol conversion between different platforms.

## Protocol Conversion

AiDog uses an **N+N canonical-hub architecture**: every incoming request is parsed into a neutral `ChatRequest` model, then re-serialized into the target platform's wire format. Conversions do not happen pairwise (N×N); they all flow through the neutral model.

```
client request (any of 5 wire protocols)
  ↓ parse_incoming_request
ChatRequest (neutral)
  ↓ convert_request
upstream request (any of 5 wire protocols) + api_path

upstream SSE (any of 5 wire protocols)
  ↓ parse_sse
ChatStreamEvent (neutral)
  ↓ to_client_sse
client SSE (matches incoming protocol)
```

### Wire protocols

Five wire protocols are supported as endpoints (the `protocol` field on each endpoint entry):

| Wire protocol | serde key | API path | Notes |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | Claude family native format |
| `OpenAI` | `openai` | `/chat/completions` | Default fallback; most aggregators |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / new OpenAI coding interface |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | Legacy text completion |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Google Gemini native |

### Conversion matrix

The neutral model guarantees every incoming protocol can be routed to every outgoing protocol. Non-streaming response conversion is currently fully implemented for **OpenAI → Anthropic** (tool-call + content coexistence fix); other cross-protocol paths fall through to upstream body passthrough at runtime.

| Incoming \ Outgoing | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | passthrough | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | passthrough | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | passthrough | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | passthrough | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | passthrough |

### Special platforms (bypass wire layer)

The following `platform_type` values are not routed through `convert_request` / `forward_attempt`. They are intercepted in `proxy/handler.rs` before the retry loop:

| platform_type | Handling |
|---|---|
| `mock` | Built-in mock adapter; no real upstream |
| `claude_code` | 1:1 passthrough of OAuth-subscribed Claude Code requests |
| `devin` | Chat ↔ session translation in `proxy/devin.rs` (create session → poll → fetch messages → wrap back to openai/anthropic) |
| `cli-proxy` | Wire / base_url / key resolved from the linked `cli_proxy_provider` row at candidate time |

## Automatic Interface Detection

AiDog detects the interface type from client request paths and bodies — no manual configuration needed:

- **Chat interface** (`/v1/chat/completions`, `/v1/messages`) — the most common AI chat requests
- **Coding assistant interface** (`/v1/responses`, Codex / Claude Code) — used by coding tools
- **Text completion interface** (`/v1/completions`) — legacy text completion requests

This means you can serve different types of clients through the same proxy address.

## Mock Platform

AiDog provides a mock platform that simulates AI responses without sending real requests. You can customize response content, status codes, delays, and more.
