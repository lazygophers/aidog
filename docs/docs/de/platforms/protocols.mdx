# Unterstützte Plattformen & Schnittstellen

AiDog unterstützt eine breite Palette von AI-Plattformen und Schnittstellentypen und übernimmt automatisch die Protokollkonvertierung zwischen verschiedenen Plattformen.

## Protokollkonvertierung

AiDog verwendet eine **N+N Canonical-Hub-Architektur**: jede eingehende Anfrage wird zunächst in ein neutrales `ChatRequest`-Modell geparst und dann in den Wire-Protokoll der Zielplattform serialisiert. Konvertierungen erfolgen nicht paarweise (N×N); alles läuft über das neutrale Modell.

```
Client-Anfrage (eines von 5 Wire-Protokollen)
  ↓ parse_incoming_request
ChatRequest (neutral)
  ↓ convert_request
Upstream-Anfrage (eines von 5 Wire-Protokollen) + api_path

Upstream-SSE (eines von 5 Wire-Protokollen)
  ↓ parse_sse
ChatStreamEvent (neutral)
  ↓ to_client_sse
Client-SSE (identisch mit eingehendem Protokoll)
```

### Wire-Protokolle

Fünf Wire-Protokolle werden als Endpunkte unterstützt (`protocol`-Feld auf jedem Endpunkt-Eintrag):

| Wire-Protokoll | serde key | API-Pfad | Hinweise |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | Natives Format der Claude-Familie |
| `OpenAI` | `openai` | `/chat/completions` | Standard-Fallback; die meisten Aggregatoren |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / neue OpenAI-Coding-Schnittstelle |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | Legacy-Text-Completion |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Natives Google Gemini |

### Konvertierungsmatrix

Das neutrale Modell garantiert, dass jedes eingehende Protokoll zu jedem ausgehenden Protokoll geroutet werden kann. Die vollständige Konvertierung von Non-Streaming-Antworten ist derzeit für **OpenAI → Anthropic** implementiert (Fix für tool_calls + content Koexistenz); andere Cross-Protocol-Pfade fallen zur Laufzeit auf Upstream-Body-Passthrough zurück.

| Eingehend \ Ausgehend | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | transparent | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | transparent | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | transparent | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | transparent | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | transparent |

### Spezielle Plattformen (umgehen die Wire-Schicht)

Die folgenden `platform_type`-Werte werden nicht über `convert_request` / `forward_attempt` geroutet. Sie werden in `proxy/handler.rs` vor der Retry-Schleife abgefangen:

| platform_type | Behandlung |
|---|---|
| `mock` | Eingebauter Mock-Adapter; kein echter Upstream |
| `claude_code` | 1:1-Passthrough von OAuth-abonnierten Claude Code-Anfragen |
| `devin` | Chat ↔ Session-Übersetzung in `proxy/devin.rs` (Session erstellen → poll → Messages abrufen → zurück nach openai/anthropic wrappen) |
| `cli-proxy` | Wire/base_url/key werden zur Kandidatenzeit aus der verknüpften `cli_proxy_provider`-Zeile aufgelöst |

## Automatische Schnittstellenerkennung

AiDog erkennt den Schnittstellentyp aus Client-Anfragepfaden und -Bodies — keine manuelle Konfiguration nötig:

- **Chat-Schnittstelle** (`/v1/chat/completions`, `/v1/messages`) — Die häufigsten AI-Chat-Anfragen
- **Coding-Assistant-Schnittstelle** (`/v1/responses`, Codex / Claude Code) — Von Coding-Tools genutzt
- **Text-Completion-Schnittstelle** (`/v1/completions`) — Legacy-Text-Completion-Anfragen

Das bedeutet, Sie können verschiedene Client-Typen über dieselbe Proxy-Adresse bedienen.

## Mock-Plattform

AiDog bietet eine Mock-Plattform, die AI-Antworten simuliert, ohne echte Anfragen zu senden. Sie können Antwortinhalt, Statuscodes, Verzögerungen u. v. m. anpassen.
