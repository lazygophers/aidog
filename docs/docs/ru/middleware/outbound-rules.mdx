# Исходящие правила

Исходящие правила выполняются **после возврата ответа от upstream, до перенаправления клиенту**. Движок применяет 3 типа исходящих правил в фиксированном порядке:

```
ResponseOverride → Rectifier → ErrorRule
```

## response_override — Переопределение ответа

Перезаписывает тела успешных ответов. Сопоставляет текст ответа по `pattern`; при совпадении заменяет содержимым из `config`. Полезно для унификации форматов полей или маскирования чувствительных фрагментов в ответах.

## rectifier — Выпрямитель

Исправляет дефекты формата в upstream-ответах:

- **Фикс SSE-потока**: добавление отсутствующих префиксов `data:`, границ событий
- **Фикс JSON**: починка усечения / нелегальных экранирований
- **Фикс кодировки**: унификация кодировки символов
- **Фикс отсутствующих полей**: заполнение обязательных полей

Используется для совместимости с upstream-платформами, возвращающими нестандартные ответы.

## error_rule — Классификация ошибок

Классифицирует ошибки upstream (не-2xx):

- **Повтор**: признан повторяемым (например 429 / 503) → запускает межплатформенный повтор отработки отказа
- **Размыкание**: признан платформенной ошибкой → засчитывается в отказы автоматического выключателя (см. [Планирование и выключатель](/ru/groups/scheduling))
- **Переопределение кода состояния**: сопоставление upstream-кода в видимый клиенту код

`config` может задавать правила `retry` / `circuit_break` / `status_override`. Набор `error_rules` по умолчанию встроен (см. [Встроенные пресеты](/ru/middleware/builtin-presets)).

## Потоковая обработка по чанкам

Для SSE-стриминга движок применяет исходящие правила **по чанкам** (`rewrite_chunk_text`) без буферизации всего ответа:

- Каждый SSE-чанк обрабатывается по поступлении
- Стриминговая семантика сохраняется; клиент получает токены в реальном времени
- Размер regex ограничен (DFA ≤ 1 МиБ) для предотвращения ReDoS
