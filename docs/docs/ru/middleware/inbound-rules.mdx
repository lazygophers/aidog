# Входящие правила

Входящие правила выполняются **до достижения запросом upstream-платформы**, действуя на тело запроса (messages, system, model и т.д.). Движок применяет 5 типов входящих правил в фиксированном порядке:

```
RequestFilter → SensitiveWord → Redaction → ContentFilter → DynamicInjection
```

## request_filter — Фильтрация полей запроса

Фильтрация запросов по значению поля. Типичное применение: **allow/deny список моделей**.

- `match_type`: метод сопоставления
- `pattern`: паттерн значения поля (например имя модели `gpt-4o`)
- `action`: `warn` (журнал) / `block` (отклонить запрос)

Пример: запретить модель → `block` при совпадении; запрос отклоняется и записывается в журнал аудита (`proxy_log.blocked_by`).

## sensitive_word — Блокировка чувствительных слов

`pattern` — само чувствительное слово (не regex). При совпадении применяется `action`:

- `block`: отклонить запрос
- `mask`: заменить плейсхолдером

Встроенные детекторы покрывают распространённые паттерны секретов (см. [Встроенные пресеты](/ru/middleware/builtin-presets)).

## redaction — Маскирование

Замена чувствительной информации в тексте запроса. Работает со встроенными regex (API-ключ, email, телефон) или пользовательским `pattern`:

- `config.replacement`: текст замены (по умолчанию `***`)
- совпадающий чувствительный фрагмент → заменяется, затем перенаправляется

## content_filter — Фильтрация содержимого

Сопоставление содержимого запроса по regex; при совпадении — блокировка или перезапись. Полезно для блокировки определённых тем или инъекции политик безопасности.

## dynamic_injection — Динамическая инъекция

Динамическое внедрение содержимого в запрос. Три точки инъекции (задаются в `config`):

- **system**: внедрить системное сообщение (например ограничение идентичности, политика безопасности)
- **header**: внедрить / переопределить HTTP-заголовки
- **body**: внедрить / переопределить поля тела запроса

Пример: автоматическое добавление системного промпта к каждому запросу → настройте `dynamic_injection` + `config.inject = system`.

## Аудит блокировок

Совпадения с `action = block` записываются в `proxy_log`:

- `blocked_by`: идентификатор сработавшего правила (`rule_type#id name`)
- `blocked_reason`: читаемая причина блокировки

Заблокированные запросы **не тарифицируются** (`est_cost = 0`), но сохраняется полная строка аудита.
