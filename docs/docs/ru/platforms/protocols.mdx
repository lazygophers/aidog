# Поддерживаемые платформы и интерфейсы

AiDog поддерживает широкий спектр AI-платформ и типов интерфейсов, автоматически обрабатывая преобразование протоколов между разными платформами.

## Преобразование протоколов

AiDog использует **N+N каноническую hub-архитектуру**: каждый входящий запрос сначала парсится в нейтральную модель `ChatRequest`, затем сериализуется в wire-протокол целевой платформы. Преобразования не выполняются попарно (N×N); всё проходит через нейтральную модель.

```
запрос клиента (один из 5 wire-протоколов)
  ↓ parse_incoming_request
ChatRequest (нейтральная)
  ↓ convert_request
upstream-запрос (один из 5 wire-протоколов) + api_path

upstream-SSE (один из 5 wire-протоколов)
  ↓ parse_sse
ChatStreamEvent (нейтральный)
  ↓ to_client_sse
клиентский SSE (совпадает с входящим протоколом)
```

### Wire-протоколы

Пять wire-протоколов поддерживаются как endpoints (поле `protocol` в каждой записи endpoint):

| Wire-протокол | serde key | API-путь | Примечания |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | Нативный формат семейства Claude |
| `OpenAI` | `openai` | `/chat/completions` | Fallback по умолчанию; большинство агрегаторов |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / новый OpenAI coding-интерфейс |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | Устаревшее text completion |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Нативный Google Gemini |

### Матрица преобразования

Нейтральная модель гарантирует, что любой входящий протокол можно направить к любому исходящему. Полное преобразование non-streaming-ответов сейчас реализовано для **OpenAI → Anthropic** (исправление сосуществования tool_calls + content); остальные cross-protocol-пути во время выполнения fallback к passthrough тела upstream.

| Входящий \ Исходящий | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | прозрачно | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | прозрачно | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | прозрачно | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | прозрачно | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | прозрачно |

### Специальные платформы (обходят wire-слой)

Следующие значения `platform_type` не маршрутизируются через `convert_request` / `forward_attempt`. Они перехватываются в `proxy/handler.rs` перед retry-циклом:

| platform_type | Обработка |
|---|---|
| `mock` | Встроенный mock-адаптер; без реального upstream |
| `claude_code` | 1:1 прозрачный passthrough OAuth-подписанных Claude Code запросов |
| `devin` | Трансляция chat ↔ session в `proxy/devin.rs` (create session → poll → fetch messages → оборачивание обратно в openai/anthropic) |
| `cli-proxy` | wire/base_url/key резолвятся из связанной строки `cli_proxy_provider` во время выбора кандидата |

## Автоопределение интерфейса

AiDog определяет тип интерфейса из путей и тел клиентских запросов — ручная настройка не требуется:

- **Чат-интерфейс** (`/v1/chat/completions`, `/v1/messages`) — наиболее частые AI chat-запросы
- **Интерфейс ассистента кода** (`/v1/responses`, Codex / Claude Code) — используется инструментами кода
- **Интерфейс text completion** (`/v1/completions`) — устаревшие запросы text completion

Это значит, что вы можете обслуживать разные типы клиентов через один адрес прокси.

## Платформа-заглушка (Mock)

AiDog предоставляет mock-платформу, симулирующую AI-ответы без реальных запросов. Можно настроить содержимое ответа, коды состояния, задержки и другое.
