# Plateformes & interfaces supportées

AiDog supporte un large éventail de plateformes AI et types d'interface, gérant automatiquement la conversion de protocole entre plateformes.

## Conversion de protocole

AiDog utilise une **architecture hub canonique N+N** : chaque requête entrante est d'abord parsée en un modèle neutre `ChatRequest`, puis re-sérialisée dans le wire protocol de la plateforme cible. Les conversions ne se font pas en pair-à-pair (N×N) ; tout transite par le modèle neutre.

```
requête client (5 wire protocols au choix)
  ↓ parse_incoming_request
ChatRequest (neutre)
  ↓ convert_request
requête amont (5 wire protocols au choix) + api_path

SSE amont (5 wire protocols au choix)
  ↓ parse_sse
ChatStreamEvent (neutre)
  ↓ to_client_sse
SSE client (identique au protocole entrant)
```

### Wire protocols

Cinq wire protocols sont supportés comme endpoints (champ `protocol` sur chaque entrée endpoint) :

| Wire protocol | serde key | API path | Notes |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | Format natif famille Claude |
| `OpenAI` | `openai` | `/chat/completions` | Fallback par défaut ; la plupart des agrégateurs |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / nouvelle interface codage OpenAI |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | Complétion de texte héritée |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Natif Google Gemini |

### Matrice de conversion

Le modèle neutre garantit que tout protocole entrant peut être routé vers tout protocole sortant. La conversion complète des réponses non-streaming est actuellement implémentée pour **OpenAI → Anthropic** (fix de coexistence tool_calls + content) ; les autres chemins cross-protocol basculent à runtime sur une transparence pass-through du body amont.

| Entrant \ Sortant | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | transparent | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | transparent | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | transparent | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | transparent | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | transparent |

### Plateformes spéciales (contournent la couche wire)

Les valeurs `platform_type` suivantes ne passent pas par `convert_request` / `forward_attempt`. Elles sont interceptées dans `proxy/handler.rs` avant la boucle de retry :

| platform_type | Traitement |
|---|---|
| `mock` | Adaptateur mock intégré ; pas d'amont réel |
| `claude_code` | Transparence 1:1 des requêtes Claude Code abonnées OAuth |
| `devin` | Traduction chat ↔ session dans `proxy/devin.rs` (create session → poll → fetch messages → wrap vers openai/anthropic) |
| `cli-proxy` | wire/base_url/key résolus depuis la ligne `cli_proxy_provider` liée au moment du candidat |

## Détection automatique d'interface

AiDog détecte le type d'interface depuis les chemins et bodies de requête client — aucune configuration manuelle nécessaire :

- **Interface chat** (`/v1/chat/completions`, `/v1/messages`) — Les requêtes de chat AI les plus courantes
- **Interface assistant de codage** (`/v1/responses`, Codex / Claude Code) — Utilisée par les outils de codage
- **Interface de complétion de texte** (`/v1/completions`) — Requêtes de complétion de texte héritées

Cela signifie que vous pouvez servir différents types de clients via la même adresse proxy.

## Plateforme mock

AiDog fournit une plateforme mock qui simule des réponses AI sans envoyer de vraies requêtes. Vous pouvez personnaliser le contenu des réponses, codes de statut, délais, et plus.
