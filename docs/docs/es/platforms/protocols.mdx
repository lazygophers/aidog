# Plataformas e Interfaces Soportadas

AiDog soporta una amplia gama de plataformas AI y tipos de interfaz, manejando automáticamente la conversión de protocolo entre diferentes plataformas.

## Conversión de protocolo

AiDog usa una **arquitectura hub canónica N+N**: cada petición entrante se parsea primero a un modelo neutro `ChatRequest`, luego se serializa al wire protocol de la plataforma destino. Las conversiones no ocurren pareja a pareja (N×N); todo fluye a través del modelo neutro.

```
petición del cliente (uno de 5 wire protocols)
  ↓ parse_incoming_request
ChatRequest (neutro)
  ↓ convert_request
petición upstream (uno de 5 wire protocols) + api_path

SSE upstream (uno de 5 wire protocols)
  ↓ parse_sse
ChatStreamEvent (neutro)
  ↓ to_client_sse
SSE cliente (idéntico al protocolo entrante)
```

### Wire protocols

Cinco wire protocols se soportan como endpoints (campo `protocol` en cada entrada endpoint):

| Wire protocol | serde key | Ruta API | Notas |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | Formato nativo familia Claude |
| `OpenAI` | `openai` | `/chat/completions` | Fallback por defecto; la mayoría de agregadores |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / nueva interfaz codificación OpenAI |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | Completado de texto heredado |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Google Gemini nativo |

### Matriz de conversión

El modelo neutro garantiza que cualquier protocolo entrante pueda rutearse a cualquier protocolo saliente. La conversión completa de respuestas non-streaming está actualmente implementada para **OpenAI → Anthropic** (fix de coexistencia tool_calls + content); otras rutas cross-protocol caen en runtime a passthrough del body upstream.

| Entrante \ Saliente | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | transparente | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | transparente | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | transparente | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | transparente | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | transparente |

### Plataformas especiales (evitan la capa wire)

Los siguientes valores `platform_type` no se rutean a través de `convert_request` / `forward_attempt`. Se interceptan en `proxy/handler.rs` antes del bucle de retry:

| platform_type | Manejo |
|---|---|
| `mock` | Adaptador mock integrado; sin upstream real |
| `claude_code` | Passthrough 1:1 de peticiones Claude Code suscritas vía OAuth |
| `devin` | Traducción chat ↔ session en `proxy/devin.rs` (create session → poll → fetch messages → envolver a openai/anthropic) |
| `cli-proxy` | wire/base_url/key se resuelven desde la fila `cli_proxy_provider` vinculada al momento del candidato |

## Detección automática de interfaz

AiDog detecta el tipo de interfaz desde rutas y bodies de peticiones del cliente — sin configuración manual:

- **Interfaz de chat** (`/v1/chat/completions`, `/v1/messages`) — Las peticiones de chat AI más comunes
- **Interfaz de asistente de codificación** (`/v1/responses`, Codex / Claude Code) — Usada por herramientas de codificación
- **Interfaz de completado de texto** (`/v1/completions`) — Peticiones de completado de texto heredadas

Esto significa que puedes servir diferentes tipos de clientes a través de la misma dirección proxy.

## Plataforma mock

AiDog proporciona una plataforma mock que simula respuestas AI sin enviar peticiones reales. Puedes personalizar el contenido de las respuestas, códigos de estado, retrasos y más.
