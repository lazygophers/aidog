# المنصات والواجهات المدعومة

يدعم AiDog طيفًا واسعًا من منصات AI وأنواع الواجهات، ويتولى تلقائيًا تحويل البروتوكول بين المنصات المختلفة.

## تحويل البروتوكول

يستخدم AiDog **بنية hub مركزية N+N**: يُوزَّع كل طلب وارد أولًا إلى نموذج `ChatRequest` محايد، ثم يُعاد تسلسله إلى بروتوكول wire للمنصة المستهدفة. التحويلات لا تحدث زوجيًا (N×N)؛ بل جميعها تمر عبر النموذج المحايد.

```
طلب العميل (أحد بروتوكولات wire الخمسة)
  ↓ parse_incoming_request
ChatRequest (محايد)
  ↓ convert_request
طلب المنبع (أحد بروتوكولات wire الخمسة) + api_path

SSE المنبع (أحد بروتوكولات wire الخمسة)
  ↓ parse_sse
ChatStreamEvent (محايد)
  ↓ to_client_sse
SSE العميل (يطابق البروتوكول الوارد)
```

### بروتوكولات wire

يتم دعم خمسة بروتوكولات wire كنقاط نهاية (حقل `protocol` على كل إدخال endpoint):

| بروتوكول wire | serde key | مسار API | ملاحظات |
|---|---|---|---|
| `Anthropic` | `anthropic` | `/v1/messages` | تنسيق عائلة Claude الأصلي |
| `OpenAI` | `openai` | `/chat/completions` | الاحتياطي الافتراضي؛ أغلب المجمّعات |
| `OpenAIResponses` | `openai_responses` | `/v1/responses` | Codex / واجهة OpenAI coding الجديدة |
| `OpenAICompletions` | `openai_completions` | `/v1/completions` | إكمال النص القديم |
| `Gemini` | `gemini` | `/v1beta/models/{model}:streamGenerateContent` | Google Gemini الأصلي |

### مصفوفة التحويل

يضمن النموذج المحايد إمكانية توجيه أي بروتوكول وارد إلى أي بروتوكول صادر. التحويل الكامل للاستجابات غير المتدفقة منفذ حاليًا بالكامل لـ **OpenAI ← Anthropic** (إصلاح تعايش tool_calls + content)؛ مسارات البروتوكولات المتقاطعة الأخرى ترجع في وقت التشغيل إلى تمرير body المنبع كما هو.

| وارد \ صادر | anthropic | openai | openai_responses | openai_completions | gemini |
|---|:---:|:---:|:---:|:---:|:---:|
| anthropic | شفاف | ✓ | ✓ | ✓ | ✓ |
| openai | ✓ | شفاف | ✓ | ✓ | ✓ |
| openai_responses | ✓ | ✓ | شفاف | ✓ | ✓ |
| openai_completions | ✓ | ✓ | ✓ | شفاف | ✓ |
| gemini | ✓ | ✓ | ✓ | ✓ | شفاف |

### منصات خاصة (تتجاوز طبقة wire)

قيم `platform_type` التالية لا تمر عبر `convert_request` / `forward_attempt`. يتم اعتراضها في `proxy/handler.rs` قبل حلقة إعادة المحاولة:

| platform_type | المعالجة |
|---|---|
| `mock` | محول mock مدمج؛ لا منبع فعلي |
| `claude_code` | تمرير شفاف 1:1 لطلبات Claude Code المشتركة عبر OAuth |
| `devin` | ترجمة chat ↔ session في `proxy/devin.rs` (create session → poll → fetch messages → لفّ إلى openai/anthropic) |
| `cli-proxy` | يتم حل wire/base_url/key من صف `cli_proxy_provider` المرتبط وقت الترشيح |

## الكشف التلقائي عن الواجهة

يكتشف AiDog نوع الواجهة من مسارات وبودي طلبات العميل — لا إعداد يدوي مطلوب:

- **واجهة الدردشة** (`/v1/chat/completions`, `/v1/messages`) — طلبات دردشة AI الأكثر شيوعًا
- **واجهة مساعد البرمجة** (`/v1/responses`, Codex / Claude Code) — تُستخدم من أدوات البرمجة
- **واجهة إكمال النص** (`/v1/completions`) — طلبات إكمال النص القديمة

هذا يعني أنه يمكنك خدمة أنواع مختلفة من العملاء عبر عنوان البروكسي نفسه.

## منصة المحاكاة

يوفر AiDog منصة محاكاة تُحاكي استجابات AI دون إرسال طلبات فعلية. يمكنك تخصيص محتوى الاستجابة، رموز الحالة، التأخيرات، والمزيد.
