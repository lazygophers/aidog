## 4. 根因（代码 + 日志双证据）

### 4.1 请求路由进入 CONNECT handler（非协议转换）

- `handler.rs:84-86`：`if req.method() == axum::http::Method::CONNECT { return super::connect::handle_connect(...).await; }` —— CONNECT 早期分流，**不走** `handle_proxy_core` 后续的组路由 / 协议转换 / 候选选择。
- 客户端配 `ALL_PROXY=http://127.0.0.1:9891/proxy` 后，任意 HTTPS 出站会先发 `CONNECT host:443 HTTP/1.1` 给代理 → 命中此 handler。
- **确认**：这两 request_id 走的是 CONNECT 隧道路径（`connect.rs`），不是 `/proxy/v1/messages` 协议转换路径。coordinator 中途修正指向正确。

### 4.2 502 终态位点（代码证据）

`connect.rs:44-56`：

```rust
let upstream = match tokio::net::TcpStream::connect(&target).await {
    Ok(s) => s,
    Err(e) => {
        tracing::warn!(error = %e, target = %target, "connect: upstream TCP failed");
        if log_enabled {
            upsert_connect_log(
                &state, request_id, String::new(), platform_id,
                target.clone(), 502, start.elapsed().as_millis() as i32,
            ).await;
        }
        return (StatusCode::BAD_GATEWAY, format!("connect {target} failed: {e}")).into_response();
    }
};
```

- 502 = **上游 TCP 连接建立失败**（`tokio::net::TcpStream::connect` 返回 Err）。
- `upstream_status_code=0` / `upstream_request_*` 全空 → 从未向上游发过任何字节，连 socket 都没建。**非上游返 5xx**。
- `duration_ms=0` → 失败极快（本地 connect 立即返回错误，典型 DNS 解析失败 / invalid addr）。

### 4.3 真根因：`target` 为空 → `TcpStream::connect("")` 必败

`connect.rs:29`：

```rust
let target = req.uri().path().to_string();
```

DB 证据（`upsert_connect_log` 字段语义锁）：

- `log.rs:274-275` 文档：`request_url = CONNECT target（host:port）`
- `connect.rs:51` 502 路径调 `upsert_connect_log(..., target.clone(), ...)` → `target` 直接入 `request_url` 列
- 测试 `test_connect.rs:64` `assert_eq!(row.request_url, "api.example.com:443")` —— **证明非空 target 会落非空 request_url**

DB 实测：6 行 `request_url` **全部长度 0**。

**双向证据闭合**：代码把 `target` 原样写 `request_url`，DB 见 `request_url=""` ⇒ **`target=""`**。

`target` 来自 `req.uri().path()`。axum 0.8 / hyper 1 对 CONNECT authority-form 请求的 URI 解析：当客户端发 `CONNECT <authority> HTTP/1.1` 时，axum 把 authority 放进 `req.uri().path()`。**`target` 为空唯一可能是 `req.uri().path()` 返回空串** —— 即客户端发的 CONNECT 请求 URI authority 段缺失或被 axum 解析掉。

`TcpStream::connect("")` 在 tokio 上立即返 `Err`（无法解析地址），秒级失败（与 `duration_ms=0` 一致），落 502。

### 4.4 链路全图

```
客户端（ALL_PROXY=127.0.0.1:9891）
  → 发 CONNECT 请求（authority-form，但 axum 解析后 .path() = ""）
  → axum fallback → handler.rs:84 命中 CONNECT
  → connect.rs:29  target = "" （req.uri().path() 返空）
  → connect.rs:44  TcpStream::connect("") → Err（地址无效）
  → connect.rs:48-53  upsert_connect_log(target="", 502, 0ms)
  → return 502 + "connect  failed: <err>" 给客户端
  → 客户端退避 ~5s 重试，6 次全失败（19:43:18 ~ 19:43:33）
```

## 7. 修复方案候选

### 候选 A（最小修，根因直击）：补 target 空兜底 + 取 authority 多源

**代价**：低（~10 行），无新依赖。
**做法**：`connect.rs:29` 改为多源取 target，空时早返 400/502 不走 connect：

```rust
// ponytail: target 多源兜底 — path() 在某些 axum/hyper 解析路径下返空，
// 补 authority().as_str() + Host header。三源皆空 = 客户端发坏请求，早返不落 connect 路径。
let target = req.uri().path().to_string();
let target = if target.is_empty() {
    req.uri().authority().map(|a| a.as_str()).unwrap_or("").to_string()
} else { target };
let target = if target.is_empty() {
    req.headers().get(axum::http::header::HOST)
        .and_then(|h| h.to_str().ok()).unwrap_or("").to_string()
} else { target };
if target.is_empty() {
    // 客户端发了无 authority 的 CONNECT —— 协议非法，不进 connect 路径
    return (StatusCode::BAD_REQUEST, "CONNECT missing target").into_response();
}
```

**风险**：若真根因不是 axum path() 解析（而是客户端真的没发 authority），三源全空 → 400，仍 6 连败但落 400 而非 502，可读性更好。需补一个 tracing::warn 把原始 `req.uri()` 整体落日志确认解析行为。
**盲区**：未实测 axum 0.8 对该客户端 CONNECT URI 的真实解析结果（DB 无原始 URI），推测成分见 §8。

### 候选 B（诊断优先，再修）：先补日志取证，确认 `req.uri()` 真实形态

**代价**：极低（3 行 tracing）。
**做法**：`connect.rs:24` 进函数即 `tracing::info!(uri = ?req.uri(), method = ?req.method(), "connect recv")`。下次复现时取真实 URI 结构，再定 A 方案的多源顺序是否正确。
**适用**：若对 A 的根因推测有疑，B 是先取证后修的稳妥路径（与 empty-stream-502.md §6 取证建议同款思路）。

### 候选 C（架构层）：评估 ALL_PROXY 路径是否应支持

**代价**：高（需求层决策）。
**背景**：P1 CONNECT 隧道设计为盲转 HTTPS（`connect.rs:1-5` 注释「P1 不解密 HTTPS，P2 才 MITM」）。当前用户用 `ALL_PROXY` 指代理，期望的是 HTTPS 出站全走代理 —— 但 P1 只盲转字节、不解密 = **看不到 apikey 无法做组路由/计费/限流**，对 aidog 核心场景（statusline 用 ANTHROPIC_BASE_URL 直连代理根）无价值。
**建议**：若用户实际只需 statusline 场景，引导用 `ANTHROPIC_BASE_URL` 直连（已工作），CONNECT 隧道仅作为兼容 HTTP_PROXY 的兜底，A/B 修法即可，不需 C。

**推荐**：**先 B 后 A**。B 一行 tracing 下次复现取证，A 多源兜底 + 空早返是稳妥修复。

## 8. Caveats / Not Found

1. **`req.uri().path()` 为何返空 — 推测成分**：DB 无原始 URI，仅由 `request_url=""` + `connect.rs:51` `target.clone()→request_url` 的字段语义反推得 `target=""`。axum 0.8 / hyper 1.10.1 对客户端 CONNECT authority-form 的解析行为（是否填 path() / 是否填 authority()）**未实测**，需候选 B 的 tracing 取证。**标 `推测:`** 的是 axum 解析路径的具体形态；`target=""` 这一事实本身由 DB + 代码双向证据闭合，非推测。

2. **客户端发的 CONNECT 目标 host 未知**：DB 无记录（request_url 空），无 tracing 落盘。可能值：`api.anthropic.com:443` / `statsig.anthropic.com:443` / 其它 Claude Code 出站域名。**6 行全空** → 倾向于「客户端发的 URI 结构 axum 没解析出 path」而非「客户端每次都发空 authority」（后者极不可能），但仍需取证。

3. **代理 tracing 日志未落盘**：`connect.rs:47` `tracing::warn!(... "connect: upstream TCP failed")` 仅写 stderr/日志面板，未入库；诊断时无该 6 次的 tracing 行可读（DB 是唯一持久化）。建议候选 B 顺带把关键诊断写 `response_body` 或 `blocked_reason` 列（log.rs:269 注释说终态一次性落，无中间节点，但 502 终态时 `response_body` 可塞错误串供取证）。

4. **6 连发无成功配对**：±60s 窗口内**仅**这 6 行 CONNECT 502，无成功隧道、无成功协议转换请求。无法判断该客户端 session 是否随后改用 `ANTHROPIC_BASE_URL` 直连成功，或持续失败。

5. **`ALL_PROXY` 环境变量值未直接取证**：coordinator 提供的 `ALL_PROXY=http://127.0.0.1:9891/proxy` 为用户陈述，DB 无 env 记录；但 `source_protocol=http-connect` 间接证实请求确实经 CONNECT 隧道路径进入代理。
